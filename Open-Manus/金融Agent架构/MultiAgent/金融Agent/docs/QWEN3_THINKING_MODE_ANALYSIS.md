# Qwen3-VL-8B-Thinking 关闭 Thinking 模式分析

## 一、Thinking 模式与关闭方式

### 1.1 模型说明

- **Qwen3-VL-8B-Thinking**：阿里推理增强的视觉语言模型，支持图文、表格处理
- **Thinking 模式**：在生成最终答案前，先输出显式的 chain-of-thought 推理，再给出答案
- **关闭方式**（视部署/API 支持情况）：
  - API 请求：`extra_body={"chat_template_kwargs": {"enable_thinking": False}}` 或 `enable_thinking: false`
  - vLLM 启动：`--model-args "use_thinking_mode=False"`

> 注：当前使用 siliconflow.cn 等国内 API，需确认是否支持 `enable_thinking` 参数；若不支持，需换用支持该参数的部署方式（如自建 vLLM）。

## 二、关闭 Thinking 对耗时的影响

### 2.1 预期收益（来自公开测试）

| 指标 | 开启 Thinking | 关闭 Thinking | 变化 |
|------|----------------|---------------|------|
| 平均延迟 | 1.2s | 0.3s | **约降 75%** |
| 响应延迟 | 基准 | 约降 29–33% | 视任务而定 |
| Token 生成量 | 基准 | 约降 15% | 减少推理 token |
| GPU 显存 | 18GB | 12GB | 约降 33% |
| 并发能力 | 基准 | 提升 10–15% | 单请求更快 |

### 2.2 在本项目中的影响

| 调用场景 | 当前耗时特征 | 关闭 Thinking 后预期 |
|----------|--------------|------------------------|
| **Planning**（多模态） | 2–4 分钟，Completion 3k–5k tokens | 可减少 30–50%，约 1–2 分钟 |
| **Multimodal 单轮** | 每轮 ~15–20s，含大量 reasoning | 单轮可降到 ~8–12s |
| **0-tool 空转** | 20 轮 × 18s ≈ 360s | 20 轮 × 10s ≈ 200s，总耗时明显下降 |
| **整体超时率** | 24/200 超时 | 有望降到 10–15/200 |

**结论**：关闭 Thinking 模式**有助于明显缩短耗时**，尤其对 Planning、Multimodal 多轮和 0-tool 空转场景。

## 三、关闭 Thinking 对任务能力的影响

### 3.1 能力变化（来自公开评测）

| 维度 | 开启 Thinking | 关闭 Thinking | 说明 |
|------|----------------|---------------|------|
| 回答质量评分 | 4.8/5.0 | 4.2/5.0 | 约降 12%，仍在可接受范围 |
| 复杂推理 | 更强 | 较弱 | 多步推理、因果分析变差 |
| 直接问答 | 略冗余 | 更直接 | 简单任务差异小 |

### 3.2 对 FinMMR 任务的影响

FinMMR 任务大致分为：

| 任务类型 | 典型特征 | 关闭 Thinking 的预期影响 |
|----------|----------|----------------------------|
| **简单表格提取** | 单表、少量数值、一步计算 | 影响小，甚至可能略快且更稳定 |
| **多表/多图关联** | 需跨图/跨表找数、多步推理 | 准确率可能下降 5–15% |
| **复杂公式** | 多变量、单位换算、条件判断 | 准确率可能下降 10–20% |
| **中文金融表格** | 表格结构复杂、需理解语义 | 准确率可能下降 5–10% |

### 3.3 与本项目问题的关系

| 现象 | 与 Thinking 的关系 | 关闭后的可能变化 |
|------|--------------------|-------------------|
| **0-tool 空转** | 模型可能先输出长段 reasoning，再输出「Thinking complete」等，未触发 tool_calls | 输出更短、更直接，**可能**减少 0-tool，但若根因是模型不返回 tool_calls，则改善有限 |
| **Planning 慢** | 规划前有大量推理 token | **明显缩短** Planning 耗时 |
| **numeric_mismatch** | 复杂推理有助于正确提取和计算 | **可能增加** 5–10% 的数值错误 |
| **tool_execution_error** | 与推理深度关系不大 | 影响较小 |

## 四、综合权衡

### 4.1 时间 vs 能力

| 策略 | 耗时 | 正确率（估计） | 适用场景 |
|------|------|----------------|----------|
| **保持 Thinking** | 高（P95≈600s，24 超时） | 66.7% 正常执行正确率 | 更看重准确率、可接受长耗时 |
| **关闭 Thinking** | 低（预计降 30–50%） | 约 60–63% | 更看重速度、可接受一定准确率损失 |
| **分场景** | 混合 | 混合 | Planning 关、Multimodal 开（若 API 支持） |

### 4.2 建议

1. **若当前主要瓶颈是耗时和超时**：可尝试关闭 Thinking，优先验证时间收益。
2. **若更看重正确率**：建议保持 Thinking，或仅对 Planning 关闭（若支持）。
3. **折中**：在 smoke 集上做 A/B 对比（开/关 Thinking），再决定是否全量关闭。

---

## 五、实施注意

1. **API 支持**：siliconflow 等国内 API 是否支持 `enable_thinking` 需实测；若不支持，需换用 vLLM 自建或支持该参数的平台。
2. **参数传递**：若支持，通常在 `extra_body` 或 `chat_template_kwargs` 中传入 `enable_thinking: false`。
3. **回归验证**：关闭后建议用 `finmmr_easy_smoke_test.json` 做回归，对比正确率和耗时。
