# 图像处理 Tool 引入可行性分析

## 一、问题陈述

**核心假设**：未处理的原始图片 + 较长固定 prompt → 复杂输入 → 长 token 输出 → 耗时长、正确率受限。

**建议**：引入专门适配多模态模型的图像处理 tool，在计划执行与数据提取前对图片进行预处理，以：
- 提升正确率
- 减少单次耗时
- 获得更高效输出

本文档从**现状**、**Token 流**、**错误类型匹配**、**可行性**、**风险**、**实施路径**六个维度深入分析。

---

## 二、现状：图片与 Token 的实际情况

### 2.1 图片输入规模（来自日志）

| 样本 | base64 长度 | 约等效 | 典型任务 |
|------|-------------|--------|----------|
| easy-test-22 | 47,616 | ~36KB 原图 | 双图 ROI 计算 |
| easy-test-0 | 112,712 | ~85KB | 财务报表 |
| easy-test-18 | 87,420 | ~66KB | 多 step 双图 |
| easy-test-11 | 303,072 | ~227KB | 复杂财务表 |
| easy-test-14 | 469,440 | ~352KB | 大表+PE |
| easy-test-29 | 200,916 | ~150KB | 诺华药物表 |

**结论**：图片 47KB–470KB 不等，大图（300KB+）占 Input 负担显著。

### 2.2 Token 瓶颈分布

| 瓶颈类型 | 典型表现 | 样本 |
|----------|----------|------|
| **Completion 主导** | 47KB 图 + 12,479 token 输出 ≈ 169s | easy-test-22 |
| **Input+Completion 双高** | 469KB 图 + 8,241 token 输出 ≈ 111s | easy-test-14 |
| **多 step 叠加** | 87KB 图 × 2 次 Multimodal ≈ 107s | easy-test-18 |

**关键发现**：
- **Completion 长度**（4.7k–12.5k token）通常是主导耗时，而非单纯图片大小。
- 小图（47KB）仍可能触发 12k token 输出（如 easy-test-22）。
- 大图（469KB）会同时加重 Input 与 Completion。

### 2.3 Vision 模型 Token 机制（简要）

- **Qwen-VL 类**：视觉 token 随分辨率变化，`num_tokens ∝ (grid_h × grid_w) / spatial_merge²`。
- **大图**：更高分辨率 → 更多 patch → 更多 Input token。
- **压缩/降分辨率**：可减少 Input token；但 **Completion 长度** 主要由任务复杂度与模型推理习惯决定，与 Input 并非简单线性关系。

## 三、错误类型与图像处理的相关性

### 3.1 按根因分类

| 错误类型 | 样本 | 图像处理是否可能帮助 | 说明 |
|----------|------|----------------------|------|
| **视觉提取失败** | easy-test-22 | ✓ 可能 | 输出 None，可能因图复杂或定位困难；预处理可裁剪/增强或先提取关键区域 |
| **单位错误** | easy-test-14 | ✓ 可能 | 26.9 vs 2.69e9；结构化 OCR 可保留单位标注 |
| **整表 dump 未选行** | easy-test-29 | ✓ 可能 | 表结构预提取可提供「行/列」索引，便于 Agent 按行选择 |
| **数据源选择** | easy-test-17 | △ 部分 | 文本 vs 表格；预提取表格可强制「先看表」 |
| **公式理解** | easy-test-2 | ✗ 否 | 语义理解问题 |
| **代码格式** | easy-test-18 | ✗ 否 | print 被注释 |

### 3.2 按任务类型

| 任务类型 | 占比 | 图像处理适用性 |
|----------|------|----------------|
| **表格提取** | 高 | 高：OCR/表结构提取、行列索引 |
| **图表数据** | 中 | 中：需图表→数据转换，精度与鲁棒性待验证 |
| **混合图文** | 中 | 中：需区分文本与表格区域 |
| **纯文本** | 低 | 无图，不适用 |

**结论**：约一半错误类型（视觉失败、单位、选行、数据源）可能受益于图像预处理；另一半（公式、代码）与图像无关。

## 四、图像处理 Tool 的形态与能力

### 4.1 可选方案

| 方案 | 能力 | 预期效果 | 成熟度 |
|------|------|----------|--------|
| **A. 压缩/降分辨率** | 减少像素、文件大小 | 降低 Input token | 高 |
| **B. OCR/文档解析** | 图→文本/结构化 Markdown | 用文本替代或补充图像 | 高 |
| **C. 表格检测+提取** | 表格区域→结构化表格 | 直接提供行列数据 | 中 |
| **D. 图表→数据** | 柱状图/折线图→数值 | 减少视觉推理 | 中 |
| **E. 区域裁剪** | 按 plan 裁剪 ROI | 减少无关区域 | 中 |
| **F. Vision Token 压缩** | 如 DeepSeek OCR 16× 压缩 | 保持布局，减少 token | 新兴 |

### 4.2 现有工具（2024–2025）

- **table-transformer**：表格检测 + 结构化提取，面向 LLM 预处理。
- **GLM-OCR / Nemotron-Parse**：轻量 OCR，文档解析。
- **DeepSeek OCR**：1024² 页→256 token，高压缩率。
- **Chart/表格专用**：如 ChartOCR、TableFormer 等。

## 五、可行性分析

### 5.1 对正确率的潜在影响

| 机制 | 预期效果 | 风险 |
|------|----------|------|
| **结构化输入** | 表格→行列数据，减少「整表 dump」与「选行错误」 | OCR 错误传播 |
| **单位保留** | 表格/文本中保留「亿」「万」等 | 解析不一致 |
| **ROI 裁剪** | 聚焦相关区域，减少干扰 | 裁剪错误丢失重要信息 |
| **简化输入** | 文本/表格替代图像 | 丢失布局、图表等视觉信息 |

**净效应**：理论上可提升约 2–4 个样本的正确率（22、14、29、17），但需严格控制 OCR 与表格解析质量。

### 5.2 对耗时的潜在影响

| 机制 | 预期效果 | 说明 |
|------|----------|------|
| **减少 Input token** | 降首 token 延迟 | 压缩/裁剪/OCR 替代 |
| **减少 Completion** | 降生成主导耗时 | 结构化输入减少「推理篇幅」 |
| **预处理开销** | 增加耗时 | OCR/表格提取约 1–5s |

**量化**：
- Input：大图 469KB 可压缩至 50–100KB 级，Input token 可能减少 30–50%。
- Completion：若结构化输入使任务更「直接」，模型可能少输出 2–4k token，对应约 50–80s 节省。
- 预处理：OCR/表格约 1–5s，需与节省的 LLM 时间权衡。

### 5.3 适用边界

| 任务 | 适用 | 不适用 |
|------|------|--------|
| 表格数值提取 | ✓ | - |
| 多图表混合 | △ 需分区域处理 | - |
| 图表→数值 | △ 需专用工具 | - |
| 布局/空间关系 | - | ✗ 需保留原图 |
| 无图样本 | - | ✗ |

## 六、风险与挑战

### 6.1 正确率风险

| 风险 | 说明 |
|------|------|
| **OCR 错误** | 数字、单位识别错误会直接导致错误答案 |
| **表格结构错** | 行列错位、合并单元格识别错误 |
| **信息丢失** | 纯文本/表格丢失图表、布局、颜色等 |
| **过度依赖** | 某些任务仍需视觉推理，纯预处理不足 |

### 6.2 流程与架构风险

| 风险 | 说明 |
|------|------|
| **调用时机** | 在 Planning 前/后？每步前？需明确 |
| **多图策略** | 多图时是分别处理还是合并？ |
| **Fallback** | 预处理失败时是否回退到原图？ |
| **与 plan 的交互** | plan 未出时如何预处理？按 plan 裁剪需 plan 先执行 |

### 6.3 成本与复杂度

- 新增工具链：OCR、表格解析、可选图表解析。
- 部署与运维：多模型/服务。
- 调参与评估：何时用预处理、何时用原图。

## 七、结论与建议

### 7.1 可行性结论

**总体可行，但需分阶段、有取舍地推进**：

1. **Token 与耗时**：当前瓶颈以 Completion 为主，Input 为辅。图像处理可同时减少 Input 与（通过结构化输入）可能减少 Completion，但需验证。
2. **正确率**：约一半错误类型可能受益；另一半与图像无关。预处理质量（OCR、表格解析）对正确率影响大。
3. **适用场景**：表格类任务最适用；图表、混合场景需额外工具与验证。

### 7.2 建议实施路径

| 阶段 | 动作 | 目标 |
|------|------|------|
| **P0 验证** | 对表格类样本做离线 OCR/表格提取，对比「原图」与「结构化输入」的 Agent 表现 | 验证正确率与耗时影响 |
| **P1 轻量** | 仅做「压缩/降分辨率」或「裁剪空白」 | 降 Input token，风险低 |
| **P2 结构化** | 引入 table-transformer 等表格提取 tool | 针对表格类任务 |
| **P3 智能路由** | 按任务类型（表格/图表/混合）选择预处理策略 | 平衡收益与风险 |

### 7.3 不建议的做法

- 全量用 OCR 替代图像：会丢失视觉信息，部分任务会变差。
- 在未验证前大规模替换：需先小规模 A/B 对比。
- 忽略 Fallback：预处理失败时必须能回退到原图。

### 7.4 与 prompt 优化的关系

- 图像处理与 prompt 优化是**正交**的：前者改输入形态，后者改指令与约束。
- 211726 的 OUTPUT 限制证明：单靠 prompt 限制输出会牺牲正确率。
- 图像处理通过**简化输入**降低模型推理负担，可能在不牺牲正确率的前提下缩短输出，与 prompt 形成互补。

## 八、小结

| 维度 | 结论 |
|------|------|
| **思路正确性** | ✓ 合理；复杂图像 + 长 prompt 确实可能导致长输出 |
| **可行性** | △ 需验证；理论上可提升正确率并降耗时，但依赖预处理质量 |
| **主要收益** | 表格类任务；减少 Input token；可能减少 Completion |
| **主要风险** | OCR/表格解析错误；信息丢失；流程与 Fallback 设计 |
| **建议** | 先做 P0 离线验证，再小规模上线轻量预处理（压缩/裁剪），最后视效果引入结构化表格提取 |
