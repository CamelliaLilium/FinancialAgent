## 2026-03-02 / 方向：Planning 工具迁移为 workflow_state

### 1) 研究动机
- 既有 `planning` 工具在语义上混合了“任务分解认知”和“状态持久化”两类职责。
- 在多智能体流程中，这会放大工具误调用与状态污染风险，降低实验可解释性。
- 目标是建立严格分层：`PlanningAgent` 负责认知规划，`workflow_state` 仅负责状态层提交。

### 2) 核心改动
- **命名与文件迁移**
  - 新增正式实现：`app/tool/workflow_state.py`
  - 兼容桥接保留：`app/tool/planning.py`（仅做别名导出，不再承载主实现）
  - 工具主名统一为 `workflow_state`，保留别名 `planning` 以兼容历史调用
- **接口导出更新**
  - `app/tool/__init__.py` 新增 `WorkflowStateTool` 导出
  - `PlanningTool` 继续导出为兼容别名
- **调用侧更新（不扩散到其它架构）**
  - `app/flow/planning.py`、`app/agent/planning.py`、`run_flow.py`、`benchmark_finben_multi_agent.py`
  - 实例化改为 `WorkflowStateTool`
- **流程约束落地**
  - Phase A 规划阶段不走工具调用，先产出 JSON 草案
  - `PlanningFlow` 通过统一提交网关 `_commit_workflow_state` 写状态
  - 去除直接写内存兜底，避免旁路状态更新

### 3) 设计假设（待实验验证）
- H1：职责解耦后，`workflow_state` 的无效/重复调用显著下降。
- H2：状态更新单入口可减少“伪完成”与不可审计状态漂移。
- H3：在简单任务上，延迟可能下降或持平；准确率至少不应因流程噪声继续下降。

### 4) 科研风险与威胁
- **内部效度风险**：模型随机性仍可能引入方差，需固定种子/批次对照。
- **构念效度风险**：仅靠标签准确率无法反映流程质量，需配合错误类型统计。
- **外部效度风险**：FinBen 文本分类结论不必然外推至复杂金融推理任务。

### 5) 建议验证方案（下一轮复跑）
- 与历史基线同配置复跑（样本、超参、模型、超时一致）。
- 对比至少三组指标：
  1. 准确率 / Macro-F1
  2. 平均耗时 / P95 耗时
  3. 错误标签分布（`repeated_tool_loop`、`same_args_repeated`、`tool_execution_error`）
- 新增人工审查子样本（建议 30 条）验证“状态轨迹是否与执行事实一致”。

### 6) 结论（当前阶段）
- 本轮改造已经把 `planning` 从“模糊工具概念”迁移为 `workflow_state` 的明确状态层能力。
- 从架构治理角度，系统由“可运行”进一步迈向“可审计、可复现、可扩展”。
- 下一阶段可在不扩散模块的前提下继续做状态转移合法性检查与实验复跑闭环。

  - 增加 `str_replace_editor` 硬门控：
    - 无 `path` / 非绝对路径 -> 阻断
    - 请求上下文无文件证据 -> 阻断
